## Chapter 1 — Fundamentals of Testing

| Keyword | Definition |
|---------|-----------|
| **coverage** | The degree to which coverage items have been exercised by a test suite |
| **debugging** | The process of finding, analysing and fixing defects |
| **defect** | An imperfection/flaw in a work product (fault/bug) |
| **error** | A human action that produces an incorrect result (mistake) |
| **failure** | An event in which a component or system does not perform the required function |
| **quality** | The degree to which a product satisfies stated and implied needs |
| **quality assurance (QA)** | Process-oriented, preventive activities to ensure good development and test processes |
| **root cause** | The fundamental reason for the occurrence of a problem |
| **test analysis** | Analysing the test basis to identify testable features — "what to test?" |
| **test basis** | The body of knowledge used to design tests |
| **test case** | A set of preconditions, inputs, actions, expected results, and postconditions |
| **test completion** | Activities at milestones to archive testware and capture lessons learned |
| **test condition** | A testable aspect of a component/system identified as a basis for test cases |
| **test control** | Taking actions to meet test objectives based on monitoring data |
| **test data** | Data needed to execute tests |
| **test design** | Elaborating test conditions into test cases — "how to test?" |
| **test execution** | Running a test and comparing actual with expected results |
| **test implementation** | Creating/acquiring testware needed for execution |
| **test monitoring** | Ongoing checking of test activities vs the plan |
| **test object** | The work product being tested |
| **test objective** | The reason or purpose for designing and running a test |
| **test planning** | Defining objectives, selecting approach, planning resources |
| **test procedure** | A sequence of test cases in execution order |
| **test process** | The set of interrelated test activities |
| **test result** | The outcome of executing a test case |
| **testing** | Activities to discover defects and evaluate quality |
| **testware** | Work products produced during the test process |
| **traceability** | Degree to which a relationship can be established between two items |
| **validation** | Checking whether the system meets users' actual needs |
| **verification** | Checking whether the system meets specified requirements |

## Chapter 2 — Testing Throughout the SDLC

| Keyword | Definition |
|---------|-----------|
| **acceptance testing** | Formal testing for readiness for deployment; validates business needs |
| **black-box testing** | Specification-based testing with no knowledge of internal structure |
| **component integration testing** | Testing interfaces and interactions between components |
| **component testing** | Testing individual components in isolation (unit testing) |
| **confirmation testing** | Testing to confirm a defect has been successfully fixed |
| **functional testing** | Testing what the system does (functions, features) |
| **integration testing** | Testing interfaces and interactions between components/systems |
| **maintenance testing** | Testing of changes to an operational system |
| **non-functional testing** | Testing how well the system behaves (quality characteristics) |
| **regression testing** | Testing to confirm no adverse consequences from a change |
| **shift left** | Performing testing earlier in the SDLC |
| **system integration testing** | Testing interfaces with other systems and external services |
| **system testing** | Testing the overall behaviour/capabilities of the entire system |
| **test level** | A group of test activities organised and managed together |
| **test type** | A group of test activities related to specific quality characteristics |
| **white-box testing** | Structure-based testing using knowledge of internal structure |

## Chapter 3 — Static Testing

| Keyword | Definition |
|---------|-----------|
| **anomaly** | Any condition deviating from expected based on specs, standards, etc. |
| **dynamic testing** | Testing that requires execution of the software |
| **formal review** | A review that follows a documented process and has formal output |
| **informal review** | A review that does not follow a defined process; no formal output |
| **inspection** | The most formal review type; follows complete generic process |
| **review** | A type of static testing where work products are examined without execution |
| **static analysis** | Analysis of work products without execution (often tool-supported) |
| **static testing** | Testing that does not require execution of the software |
| **technical review** | Performed by technically qualified reviewers; focuses on technical issues |
| **walkthrough** | A review led by the author to educate and detect anomalies |

## Chapter 4 — Test Analysis and Design

| Keyword | Definition |
|---------|-----------|
| **acceptance criteria** | Conditions an implementation must meet to be accepted by stakeholders |
| **acceptance test-driven development (ATDD)** | Test-first approach where tests derived from acceptance criteria before development |
| **black-box test technique** | Technique based on specified behaviour without knowledge of internals |
| **boundary value analysis (BVA)** | Technique exercising values at the boundaries of equivalence partitions |
| **branch coverage** | % of branches in code exercised by test cases |
| **checklist-based testing** | Experience-based technique using a checklist to derive tests |
| **collaboration-based test approach** | Focus on defect avoidance via collaboration and communication |
| **coverage** | Degree to which coverage items have been exercised |
| **coverage item** | An attribute required to be exercised to satisfy a coverage criterion |
| **decision table testing** | Black-box technique using a table of condition combinations and actions |
| **equivalence partitioning (EP)** | Divides data into partitions where all elements are processed the same way |
| **error guessing** | Experience-based technique anticipating errors based on tester knowledge |
| **experience-based test technique** | Uses tester knowledge and experience to design tests |
| **exploratory testing** | Tests simultaneously designed, executed, and evaluated while learning |
| **state transition testing** | Tests based on states and valid/invalid transitions in a state model |
| **statement coverage** | % of executable statements in code exercised by test cases |
| **test technique** | A procedure used to define test conditions, test cases, and test data |
| **white-box test technique** | Technique based on analysis of the test object's internal structure |

## Chapter 5 — Managing Test Activities

| Keyword | Definition |
|---------|-----------|
| **defect management** | Process for handling anomalies from discovery to closure |
| **defect report** | Documentation of a defect and its context |
| **entry criteria** | Conditions that must be met before starting an activity |
| **exit criteria** | Conditions that must be met to declare an activity complete |
| **product risk** | Risk related to product quality characteristics |
| **project risk** | Risk related to management and control of the project |
| **risk** | A potential event whose occurrence causes an adverse effect |
| **risk analysis** | Risk identification + risk assessment |
| **risk assessment** | Determining risk likelihood, impact, and level; prioritising |
| **risk control** | Risk mitigation + risk monitoring |
| **risk identification** | Generating a comprehensive list of risks |
| **risk level** | Measure of risk based on likelihood and impact |
| **risk management** | All activities to manage risk (analysis + control) |
| **risk mitigation** | Implementing actions to reduce the risk level |
| **risk monitoring** | Ensuring mitigation is effective; identifying emerging risks |
| **risk-based testing** | Test approach based on risk analysis and control |
| **test approach** | Implementation of the test strategy for a specific project |
| **test completion report** | Summary report prepared at end of test activity |
| **test control** | Actions taken to meet test objectives based on monitoring data |
| **test monitoring** | Ongoing checking of test activities against the test plan |
| **test plan** | Document describing the test objectives, resources, and processes |
| **test planning** | Activity of establishing/updating a test plan |
| **test progress report** | Report produced during test monitoring at regular intervals |
| **test pyramid** | Model showing tests at different granularity levels |
| **test strategy** | Generalised description of the test process at a high level |
| **testing quadrants** | Model grouping test types/levels for Agile planning |
