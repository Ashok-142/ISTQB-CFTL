## 4.1 Test Techniques — Overview

A **test technique** is a method used to define test conditions, test cases, and test data. The ISTQB syllabus groups all test techniques into three categories:

| Category | Based On | Key Idea | When to Use |
|---|---|---|---|
| **Black-box** | Specification / Requirements | Test WHAT the system should do — no access to internal code needed | Requirements are available; testing from user perspective |
| **White-box** | Internal Code / Structure | Test HOW the system works — tests every path, branch, or statement | Source code is accessible; measuring code coverage |
| **Experience-based** | Tester's Knowledge & Intuition | Tester uses past experience and gut feeling to find defects | Specs are incomplete or missing; complementing other techniques |

> 💡 **Exam Tip:** Experience-based testing can be used alongside Black-box OR White-box testing — it complements them, it does not replace them.

> 💡 **Exam Tip:** Look for the KEY CLUE in the scenario: `requirements document` → Black-box. `source code / branches` → White-box. `no spec, relies on intuition` → Experience-based.

## 4.2 Black-box Techniques

### Technique 1 — Equivalence Partitioning (EP)

**Core idea:** Group all possible inputs into partitions where every value in a group is expected to behave the same way. Test one representative value from each partition.

#### Rules:
- Identify valid partitions (inputs the system accepts)
- Identify invalid partitions (inputs the system rejects)
- Pick **ONE** representative value from each partition
- Number of test cases = number of partitions

#### Example — ATM withdrawal (₹100 to ₹10,000):

| Partition | Range | Representative Value | Expected Result |
|---|---|---|---|
| Too small (invalid) | Below ₹100 | ₹50 | ❌ Rejected |
| Valid | ₹100 – ₹10,000 | ₹500 | ✅ Accepted |
| Too large (invalid) | Above ₹10,000 | ₹15,000 | ❌ Rejected |

> 💡 **Exam Tip:** EP gives you the MINIMUM number of test cases. Always check you have both valid AND invalid partitions!

---

### Technique 2 — Boundary Value Analysis (BVA)

**Core idea:** Bugs are most likely to occur at the EDGES of partitions (off-by-one errors are very common). Test the value just below, exactly at, and just above each boundary.

#### Rules:
- Identify all boundaries (transition points between partitions)
- For each boundary: test **(boundary − 1)**, **boundary**, **(boundary + 1)**
- Number of test cases = number of boundaries × 3
- Remove duplicates if boundary values overlap

#### Example — ATM withdrawal (₹100 to ₹10,000):

| Boundary | Below (−1) | At boundary | Above (+1) |
|---|---|---|---|
| Lower boundary at ₹100 | ₹99 | ₹100 | ₹101 |
| Upper boundary at ₹10,000 | ₹9,999 | ₹10,000 | ₹10,001 |

**Total BVA test cases:** 2 boundaries × 3 = **6 test cases**

> 💡 **Exam Tip:** BVA BUILDS ON EP — identify partitions first, then test the edges. BVA coverage includes EP coverage automatically.

#### Formula Summary:

| Technique | Formula | ATM Example |
|---|---|---|
| Equivalence Partitioning | 1 test case per partition | 3 partitions → 3 test cases |
| Boundary Value Analysis | Boundaries × 3 (remove duplicates) | 2 boundaries × 3 → 6 test cases |

### Technique 3 — Decision Table Testing

**Core idea:** When a system's output depends on combinations of multiple conditions, test EVERY possible combination. No combination should be left untested.

#### Rules:
- Identify all conditions (each must be Yes/No or True/False)
- Number of combinations = **2ⁿ** (where n = number of conditions)
- Each combination = one test case
- Fill in the expected outcome for each combination

#### Quick reference — 2ⁿ formula:

| No. of Conditions (n) | 2ⁿ | Test Cases Needed |
|---|---|---|
| 1 | 2¹ | 2 |
| 2 | 2² | 4 |
| 3 | 2³ | 8 |
| 4 | 2⁴ | 16 |

#### Example — Pizza discount (Member? + Order above ₹500?):

| # | Member? | Above ₹500? | Discount |
|---|---|---|---|
| 1 | Yes | Yes | 20% |
| 2 | Yes | No | 10% |
| 3 | No | Yes | 5% |
| 4 | No | No | 0% |

> 💡 **Exam Tip:** Decision Table is ideal when multiple conditions interact to produce different outputs. Always check ALL 2ⁿ combinations are covered!

## Choosing the Right Technique — Quick Guide

| Scenario clue | Best Technique |
|---|---|
| Input has a numeric range (age, amount, score) | EP + BVA |
| Multiple yes/no conditions affecting the outcome | Decision Table |
| System moves through states (login → locked → unlocked) | State Transition Testing |
| No spec available, tester uses gut feeling | Experience-based |
| Source code / branches / paths available | White-box |
| Two techniques apply to different parts of the same system | Combine techniques! |

---

## Exam-Day Checklist ✅

- For **EP**: Always include BOTH valid and invalid partitions
- For **BVA**: Test value BELOW, AT, and ABOVE each boundary
- For **BVA**: Count boundaries first, then multiply by 3
- For **Decision Table**: Use 2ⁿ formula to verify test case count
- **Experience-based** can ALWAYS combine with other techniques
- Look for the **key clue** in the scenario to identify the technique

> 💡 **Exam Tip:** In the ISTQB exam, read the scenario carefully for clues like "requirements document", "source code", "tester experience", "range of values", or "combination of conditions".
